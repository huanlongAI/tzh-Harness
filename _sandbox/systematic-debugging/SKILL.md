## 1. Purpose（目的）

提供结构化的 Bug 修复流程，确保 Agent 在修复任何技术问题时都先找到根因（Root Cause），而不是盲目猜测和打补丁。

**Non-goals**：
- 不替代领域特定的调试工具或知识
- 不适用于需求澄清类问题（那是 Step 0 Brainstorming 的职责）

## 2. Trigger（触发条件）

**使用本 Skill 的场景**：
- 测试失败（Test Failure）
- 生产 Bug（Production Bug）
- 行为与预期不符（Unexpected Behavior）
- 性能问题（Performance Issue）
- 构建失败（Build Failure）
- 集成问题（Integration Issue）

**尤其在以下情况强制使用**：
- 时间紧迫（越急越不能猜）
- 「一个小修复就行」看起来显而易见
- 已经尝试过 ≥1 次修复但没解决

**不使用的场景**：
- 需求不清晰（→ Step 0 Brainstorming）
- 纯配置变更且有文档（→ 按文档操作）

## 3. Iron Law（铁律）

<HARD-GATE>
未完成 Phase 1 根因调查之前，禁止提出任何修复方案。禁止写任何修复代码。禁止说「我觉得问题是……让我先修一下试试」。
</HARD-GATE>

## 4. Workflow — 四阶段流程

你**必须**按顺序完成每个阶段，不得跳过。

### Phase 1｜根因调查（Root Cause Investigation）

**在尝试任何修复之前**：

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 完整阅读 stack trace（堆栈追踪）
   - 记录行号、文件路径、错误码

2. **稳定复现**
   - 能否可靠触发？精确步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜

3. **检查近期变更**
   - `git diff`、最近 commits
   - 新依赖、配置变更、环境差异

4. **多组件系统：逐层收集证据**
   - 在每个组件边界添加诊断日志
   - 记录数据进入和离开每个组件的状态
   - 运行一次收集证据 → 定位断裂层 → 再深入调查

5. **根因回溯（Root Cause Tracing）**
   - 从错误点开始反向追踪调用链
   - 在每一层问：「是谁用什么值调用了这里？」
   - 一直追溯到源头
   - **在源头修复，不在症状处修复**

### Phase 2｜模式分析（Pattern Analysis）

1. **找到可工作的同类代码**——同一代码库中有什么类似的能正常运行？
2. **完整阅读参考实现**——不要跳读，逐行理解
3. **列出所有差异**——工作版本 vs 出问题版本的每一个区别
4. **理解依赖关系**——配置、环境、前置假设

### Phase 3｜假设与验证（Hypothesis & Testing）

1. **提出单一假设**：「我认为根因是 X，因为 Y」
2. **最小化测试**：做最小的变更来验证假设，一次只改一个变量
3. **验证结果**：
   - 成功 → Phase 4
   - 失败 → 提出**新**假设，**不要**在旧修复上叠加
4. **不确定时坦承**：「我不理解 X」，不要假装知道

### Phase 4｜实施修复（Implementation）

1. **创建失败测试用例**——在修复前必须有
2. **实施单一修复**——只针对已确认的根因，一次只改一处
3. **验证修复**——测试通过？没有引入新的失败？
4. **≥3 次修复失败 → 停下，质疑架构**

<HARD-GATE>
如果已尝试 3 次以上修复均未解决问题，必须立即停止，将情况升级给创始人讨论架构问题。禁止尝试第 4 次修复。
</HARD-GATE>

**3+ 次失败的信号**：
- 每次修复揭示了不同位置的新问题
- 修复需要「大规模重构」才能实施
- 每次修复在别处产生新症状

→ 这不是假设失败，这是**架构错误**。

## 5. 辅助技术

### 5.1 纵深防御（Defense-in-Depth）

修复 Bug 后，在数据流经的每一层都添加验证：

| 层 | 目的 | 示例 |
|---|------|------|
| Layer 1: 入口验证 | 拒绝明显无效输入 | 参数非空/存在/可写检查 |
| Layer 2: 业务逻辑验证 | 确保数据对当前操作有意义 | 状态一致性检查 |
| Layer 3: 环境守卫 | 防止特定上下文的危险操作 | 测试环境禁止写入源码目录 |
| Layer 4: 调试桩 | 为未来排查提供上下文 | 关键操作前记录 stack trace |

**单点验证 = 「我们修了 Bug」；四层验证 = 「我们让这个 Bug 不可能再发生」。**

### 5.2 条件等待（Condition-Based Waiting）

替换所有任意延时为条件轮询：

```
❌ await sleep(500)  // 猜测耗时
✅ await waitFor(() => result !== undefined)  // 等待实际条件
```

**只有在以下情况才允许使用固定延时**：
1. 先等待了触发条件
2. 延时基于已知计时（非猜测）
3. 注释说明了为什么

## 6. Anti-Rationalization（反合理化）

| 借口 | 现实 |
|------|------|
| 「问题很简单，不需要走流程」 | 简单问题也有根因，流程对简单 Bug 更快 |
| 「紧急情况，没时间走流程」 | 系统化调试比猜测试错快得多 |
| 「先试一下这个，不行再调查」 | 第一次修复就定型了方向，从一开始就做对 |
| 「先修好再补测试」 | 没有测试的修复不可靠，先测试才能证明 |
| 「一次改多处节省时间」 | 无法隔离哪个有效，还会引入新 Bug |
| 「参考文档太长，我大概看看就行」 | 半吊子理解一定出 Bug，完整阅读 |
| 「我看到问题了，让我修一下」 | 看到症状 ≠ 理解根因 |
| 「再试一次」（已失败 2+ 次）| 3+ 次失败 = 架构问题，别再修了 |

### Red Flags 自检

如果你发现自己在想以下任何念头，**立即停止，回到 Phase 1**：
- 「先快速修一下，以后再调查」
- 「试着改一下 X 看看行不行」
- 「一次加多个改动，跑一下测试」
- 「跳过测试，我手动验证」
- 「大概是 X 的问题，我先修了」
- 「我不太确定但这个可能管用」
- 「文档说要 X 但我改造一下」
- 「把主要问题列出来」（还没调查就列修复方案）
- 「再来一次」（已经试了 2+ 次）

## 7. Verification（验证）

**DoD 条目**：
- [ ] Phase 1 完成：根因已识别并记录
- [ ] Phase 2 完成：已找到可工作的对比参照
- [ ] Phase 3 完成：假设已验证
- [ ] Phase 4 完成：修复已实施，测试通过
- [ ] 无新测试失败引入
- [ ] （推荐）纵深防御验证层已添加

## 8. Quick Reference（速查）

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| Phase 1 根因调查 | 读错误、复现、查变更、收集证据 | 理解 WHAT 和 WHY |
| Phase 2 模式分析 | 找可工作示例、对比差异 | 识别出差异点 |
| Phase 3 假设验证 | 形成理论、最小化测试 | 假设确认或替换 |
| Phase 4 实施修复 | 写测试、修复、验证 | Bug 修复，测试通过 |

## 9. Change Policy

- Patch：示例增补、措辞修订
- Minor：新增辅助技术、新增反合理化条目
- Major：改变四阶段结构或 Iron Law（需 ADR）
